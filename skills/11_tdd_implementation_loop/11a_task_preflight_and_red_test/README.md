# 11a Task Preflight and Red Test

## 1. What this sub-SKILL is for

This reusable sub-SKILL is part of Stage 11 `11_tdd_implementation_loop`.

Purpose: Validates one approved task, inspects relevant code and tests, and creates or identifies the initial failing test.

It is an internal execution unit. Downstream stages should not depend on this sub-SKILL directly.

## 2. When to use it

Use this sub-SKILL only within the parent Stage 11 package for a selected `TASK_ID`.

Run it after: `/skills/11_tdd_implementation_loop/SKILL.md`.

Run next: `/skills/11_tdd_implementation_loop/11b_minimal_implementation_green/SKILL.md`.

## 3. When not to use it

Do not use this sub-SKILL when:

- no stable `TASK_ID` is selected;
- required approved task or prompt inputs are missing;
- the task scope is unclear or conflicts with approved decisions;
- the requested work belongs to another stage;
- using this sub-SKILL would skip the parent stage contract or finalizer.

Common Stage 11 inputs:

- `/skills/11_tdd_implementation_loop/SKILL.md`

Sub-SKILL-specific prior outputs:

## 5. Optional or conditional inputs

Read previous-stage artifacts only when the selected task touches that area:

- requirements and acceptance criteria;
- domain model and invariants;
- architecture, module boundaries, API contracts, or integration contracts;
- data schema, migrations, security rules, or privacy rules;
- Stage 12 review notes if the task fixes a review finding.

## 6. Outputs created when executed

Mandatory outputs:
- `/workflow/11_implementation_results/11_test_evidence_{{TASK_ID}}.md` — Red-step test evidence or test-aware validation plan
- `/workflow/11_implementation_results/11_task_result_{{TASK_ID}}.md` — Initial task result draft with preflight findings and scope restatement

Conditional outputs:
- None unless activated by the parent SKILL, task card, or implementation prompt.

## 7. Human approval requirements

This sub-SKILL may prepare evidence, findings, or a partial result, but it does not approve the task. Human approval is handled through the Stage 11 finalizer and subsequent review.

## 8. How to run this sub-SKILL in an Agentic Coding tool

1. Open this `SKILL.md` after confirming the parent Stage 11 package is being used.
2. Provide or confirm the selected `TASK_ID`.
3. Let the agent read only the required inputs and task-relevant code/tests.
4. Review the generated output before running the next sub-SKILL when the task is high-risk.

## 10. Draft / Approved / Needs Review

- `Draft`: Generated by the agent and not reviewed.
- `Needs Review`: Ready for human review or next sub-SKILL inspection.
- `Approved`: Explicitly approved by the human developer.
- `Blocked`: Cannot proceed safely without human decision.

## 11. Common mistakes

- Running this sub-SKILL outside the parent Stage 11 package.
- Treating partial output as final task approval.
- Skipping prerequisite sub-SKILL outputs.
- Expanding implementation scope beyond the approved task.
- Claiming validation success without recorded evidence.

## 12. Troubleshooting / blocker cases

Return a blocker report if required inputs are missing, code cannot be inspected, validation cannot be run, scope conflicts exist, or the task requires an unapproved technical or risk decision.

## 13. Recommended next step

After this sub-SKILL completes successfully, run `11b_minimal_implementation_green` unless the output is blocked or the human requests correction.
