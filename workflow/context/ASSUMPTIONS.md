# Working Assumptions

본 프로젝트 수행과 관련하여 설정된 현재의 작업 가정 사항들입니다. 인간 개발자의 피드백을 통해 검증 및 보완될 예정입니다.

## 1. 프로젝트 성격 및 유형 관련
- **A-101**: 본 프로젝트는 Greenfield 신규 개발이며, 기존의 소스 코드 베이스나 레거시 의존성이 전혀 존재하지 않는 상태에서 시작합니다.
- **A-102**: 프로젝트의 주요 목적은 콜센터 유해 발화 탐지 연구를 위한 데이터 관리 및 다중 평가(Human, LLM) 비교 분석 도구 구축입니다. 이에 따라, 화려한 일반 웹 서비스 기능보다는 연구 통계와 라벨링 편의성, 그리고 연구 무결성에 집중합니다.

## 2. 기술 스택 및 아키텍처 관련
- **A-201**: 클라이언트 중심의 React 웹 애플리케이션으로 동작하며, 백엔드 서버(Node, Python 등)를 별도로 운영하지 않고 Firebase Client SDK(Auth, Firestore, Hosting)를 직접 호출하는 classical serverless 형태로 빌드합니다.
- **A-202**: UI 스타일링은 Vanilla CSS를 사용하여 rich한 시각 효과(Glassmorphism, 다크 모드, 카드형 레이아웃)를 부여하고, 반응형 레이아웃을 통해 모바일/태블릿/데스크톱 모두를 지원합니다.

## 3. 데이터 및 보안 관련
- **A-301**: 실제 상담 대화 데이터셋은 시스템 외부의 별도 작업으로 비식별화(Anonymization) 처리가 완료된 상태로만 입력 및 등록됩니다.
- **A-302**: 평가자(상담원, 언어학자, 연구원)의 수는 10명 내외의 소규모 그룹이므로 동시성 트래픽 이슈나 복잡한 다중 조직(multi-tenancy) 관리는 불필요합니다.
- **A-303**: LLM 판정은 별도의 외부 시스템에서 독립적으로 수행되며, 그 결과(JSON)를 일괄 업로드하여 Firestore에 영구 저장하고 전문가 라벨과 매핑/비교하는 방식을 사용합니다.
- **A-304**: Kappa 등의 복잡한 다중 평가자 간의 신뢰도/합의도 통계 연산은 시스템 외부에서 별도로 수행되며, 본 시스템은 그 결과 통계를 업로드받아 화면(Dashboard)에 시각화해 보여주는 역할에 집중합니다.
- **A-104**: 평가를 수행하는 총인원은 10명 내외이므로, 동시성 충돌이나 정교한 Lock 메커니즘 없이 단순 Firestore Document 업데이트 방식으로 처리해도 데이터 무결성에 큰 지장이 없을 것으로 가정함.
- **A-305**: 외부 비식별화 검수가 사전에 완벽히 이루어지므로, 본 시스템에서는 별도의 대화 텍스트 디코딩/복호화 기능이 불필요하다고 가정함.

## 4. 도메인 모델링 및 비즈니스 규칙 관련
- **A-401**: 동일 `call_id` 대화 데이터가 재업로드될 때 기존에 누적 작성된 평가 라벨(`Label`) 및 외부 모델 데이터의 유실을 차단하기 위해, 일방적인 덮어쓰기(Overwrite)를 배제하고 무시(Skip) 혹은 기 입력 데이터를 보존하며 병합하는 멱등성 보호 처리를 도메인 기본 동작으로 가정합니다.
